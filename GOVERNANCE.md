# Verascope Governance

Verascope is currently maintained by its founding maintainer with community
contributors. The project aims to stay small, auditable, and clear about what
provenance verification can establish.

## How decisions are made

The maintainer reviews issues and pull requests against the project principles:

1. C2PA verification and heuristic signals remain structurally separate.
2. User-facing language must not make real/fake or AI/non-AI claims.
3. Core verification stays fully local and requires no network access.
4. Changes should improve correctness, clarity, accessibility, or sustainable
   maintenance without expanding the product into a moderation platform.

For a substantial feature, open an issue first. Small bug fixes, tests, and
documentation improvements can go straight to a pull request.

## Becoming a maintainer

Regular contributors may be invited to help maintain an area after showing
sound technical judgment, constructive review habits, and consistent respect
for the principles above. Initial responsibility is scoped to a specific area,
such as tests, releases, documentation, or accessibility.

Maintainers are expected to:

- Review contributions with the same evidence standard applied to their own.
- Protect the offline and calibrated-language guarantees.
- Explain decisions respectfully and keep discussions public where possible.
- Avoid merging their own substantial pull requests without another review.

Access is expanded gradually and may be removed if it is unused or if these
responsibilities are not upheld.

## Community channels

Use GitHub Issues for reproducible bugs and scoped proposals. Use pull-request
reviews for implementation discussion. Follow the [Code of
Conduct](CODE_OF_CONDUCT.md) in every project space.
